### 1️⃣ 보안 전문가 관점

#### ✅ 잘한 점
- **RLS 정책 적용**: payments 테이블에 Row Level Security 완벽 적용
- **서버 측 검증**: 결제 승인을 서버 액션에서만 처리
- **환경 변수 분리**: API 키를 `.env.local`에 안전하게 보관
- **에러 로깅**: 민감 정보 노출 없이 디버깅 가능한 로그

#### ⚠️ 개선 필요
1. **결제 금액 검증 부족**
   - **문제**: 클라이언트에서 전달된 `amount`를 그대로 신뢰
   - **위험**: 악의적 사용자가 금액 조작 가능
   - **해결책**: 서버에서 `credits` 기반으로 금액 재계산 필요
   
   ```typescript
   // 현재 (취약)
   const amount = Number(searchParams.get("amount"));
   
   // 개선안
   const credits = Number(searchParams.get("credits"));
   const PRICE_MAP = { 1: 9900, 3: 24900, 5: 39900 };
   const amount = PRICE_MAP[credits];
   if (!amount) throw new Error("Invalid credits");
   ```

2. **Rate Limiting 미적용**
   - **문제**: 무한 결제 요청 가능
   - **해결책**: Vercel Edge Config 또는 Redis로 Rate Limiting 구현

3. **결제 중복 방지 미흡**
   - **문제**: 같은 `orderId`로 여러 번 승인 시도 가능
   - **해결책**: DB에 `orderId` UNIQUE 제약 조건 추가 (✅ 이미 적용됨)

#### 🎯 보안 점수: 7/10
- **강점**: RLS, 서버 액션, 환경 변수 관리
- **약점**: 금액 검증, Rate Limiting

### 2️⃣ UX/디자인 전문가 관점

#### ✅ 잘한 점
- **Glassmorphism 디자인**: 프리미엄하고 현대적인 느낌
- **로딩 상태 표시**: 모든 비동기 작업에 로딩 인디케이터
- **에러 핸들링**: 사용자 친화적인 에러 메시지
- **CreditBalance 위젯**: 실시간 잔액 표시로 투명성 확보

#### ⚠️ 개선 필요
1. **결제 프로세스 가시성 부족**
   - **문제**: 사용자가 현재 어느 단계인지 모름
   - **해결책**: 단계별 프로그레스 바 추가
   
   ```
   [1. 정보 입력] → [2. 결제] → [3. 분석 중] → [4. 완료]
   ```

2. **결제 실패 시 복구 경로 불명확**
   - **문제**: PaymentFailPage에서 "다시 시도" 버튼이 어디로 가는지 모호
   - **해결책**: 명확한 CTA와 이전 입력 데이터 복원

3. **크레딧 충전 후 피드백 부족**
   - **문제**: 충전 성공 시 축하 애니메이션이나 명확한 알림 없음
   - **해결책**: Confetti 애니메이션 + 토스트 알림 강화

4. **모바일 최적화 미흡**
   - **문제**: PaymentWidget의 카드 레이아웃이 작은 화면에서 답답함
   - **해결책**: 모바일에서는 세로 스크롤 레이아웃으로 변경

#### 🎯 UX 점수: 8/10
- **강점**: 프리미엄 디자인, 명확한 피드백
- **약점**: 프로세스 가시성, 모바일 최적화

### 3️⃣ 시니어 엔지니어 관점

#### ✅ 잘한 점
- **타입 안전성**: TypeScript 100% 활용
- **에러 핸들링**: try-catch + 명시적 throw
- **코드 재사용성**: Server Actions 분리
- **데이터베이스 설계**: 정규화 + 인덱스 최적화

#### ⚠️ 개선 필요
1. **트랜잭션 처리 부족**
   - **문제**: 결제 승인 → 크레딧 저장 → 분석 시작이 원자적이지 않음
   - **위험**: 중간에 실패 시 데이터 불일치
   - **해결책**: Supabase의 `rpc`로 트랜잭션 구현
   
   ```typescript
   // 개선안: 하나의 트랜잭션으로 묶기
   await supabase.rpc('process_payment_and_analysis', {
     payment_data: {...},
     analysis_data: {...}
   });
   ```

2. **에러 복구 전략 미흡**
   - **문제**: 결제는 성공했지만 분석 시작 실패 시 크레딧만 차감됨
   - **해결책**: 
     - 분석 실패 시 크레딧 복구 로직
     - 재시도 큐 시스템 (Vercel Cron Jobs)

3. **성능 최적화 여지**
   - **문제**: `getAvailableCredits()`가 매번 DB 조회
   - **해결책**: Redis 캐싱 또는 React Query로 클라이언트 캐싱

4. **테스트 코드 부재**
   - **문제**: 단위 테스트, 통합 테스트 없음
   - **해결책**: Jest + React Testing Library 도입

5. **로깅 및 모니터링 부족**
   - **문제**: 프로덕션 에러 추적 어려움
   - **해결책**: Sentry 또는 Vercel Analytics 연동

#### 🎯 엔지니어링 점수: 7/10
- **강점**: 타입 안전성, 코드 구조
- **약점**: 트랜잭션, 테스트, 모니터링

## 💡 배운 점 (Lessons Learned)

### 1. 데이터베이스 스키마 불일치의 위험성
- **문제**: 기존 테이블과 코드의 컬럼명 불일치로 50분 소요
- **교훈**: 마이그레이션 스크립트를 먼저 작성하고 코드 작성
- **적용**: 앞으로 모든 테이블 변경은 SQL 파일로 관리

### 2. 에러 로깅의 중요성
- **문제**: 초기에 에러를 조용히 무시해서 디버깅 어려움
- **교훈**: 모든 에러는 명시적으로 throw하고 로그 남기기
- **적용**: `console.error` + `throw new Error` 패턴 표준화

### 3. 클라이언트 vs 서버 검증
- **문제**: 클라이언트에서 전달된 데이터를 서버에서 재검증 안 함
- **교훈**: "클라이언트는 믿을 수 없다" 원칙 철저히
- **적용**: 모든 Server Action에서 입력 검증 필수

### 4. Hydration 안전성의 중요성
- **문제**: Phase 6에서 해결했지만 새 컴포넌트에서 재발 가능
- **교훈**: 모든 클라이언트 컴포넌트에 `isMounted` 패턴 적용
- **적용**: 컴포넌트 템플릿 생성 시 기본 포함
